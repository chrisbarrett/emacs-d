## Context

`openspec/specs/gfm-pretty/spec.md:851` (Requirement: Code-fence bordered-block rendering) describes the body-line decoration as a `│ ` wrap-prefix. The actual implementation uses two distinct strings:

- A body-line `before-string` of `│ ` (`lisp/gfm/gfm-pretty-fences.el:329`), painted by an overlay covering each body line — visible on every body line.
- A separate `wrap-prefix` glyph (`↪ ` as of 2026-05-20, previously `⋱ `) generated by `gfm-pretty--wrap-prefix` (`lisp/gfm/gfm-pretty-borders.el:187`) — visible only on continuation visual rows produced by word-wrap.

Callouts differ: their wrap-prefix really is `│ ` (the left edge of the box continues onto wrapped rows), so callouts spec language at `gfm-pretty/spec.md:631` is correct and stays.

## Goals / Non-Goals

**Goals:**

- Reword the fences requirement so the body before-string and the continuation wrap-prefix are described as separate concerns.
- Keep the spec implementation-agnostic about the specific continuation glyph — `↪ ` happens to be the current choice but the requirement should not pin it.

**Non-Goals:**

- No code changes. Implementation already matches the corrected description.
- No edit to the callouts requirement; it is already accurate.
- No new requirements; the change is a clarification of an existing one.

## Decisions

**Decision: Treat as a spec delta, not a hygiene-only edit.**

The edit touches the bullet list inside a `### Requirement:` body, so per `openspec/rules/openspec-specs.md` rule 3 it does not qualify as pure hygiene and needs a spec delta under `specs/gfm-pretty/spec.md`.

**Decision: Leave the wrap-prefix glyph unspecified.**

Naming `↪ ` (or any other glyph) in the requirement body would force a spec change every time the visual is tweaked. Reference the glyph descriptively ("a continuation glyph on visual continuation rows") and let the implementation pick.

Alternative considered: pin `↪ ` in the spec. Rejected — couples the requirement to a styling choice that has already been changed once (`⋱` → `↪`).

## Risks / Trade-offs

- [Risk] A future reader sees the rule-3 carve-out and treats every requirement-body bullet edit as hygiene-only.
  → Mitigation: the proposal explicitly states this edit *is* inside a requirement body, so the delta path is documented.
